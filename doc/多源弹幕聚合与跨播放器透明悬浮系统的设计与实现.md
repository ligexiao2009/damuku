# 多源弹幕聚合与跨播放器透明悬浮系统的设计与实现

## 摘要

弹幕（Danmaku）是东亚视频文化的核心交互形式，将用户评论以水平滚动方式实时叠加于视频画面之上，形成"虚拟共视"的社交体验。然而，主流视频平台的弹幕系统均为封闭生态——Bilibili、腾讯视频、芒果TV 等平台的弹幕数据被锁定在各自的网页或客户端播放器内，用户使用 IINA、PotPlayer 等第三方播放器观看本地视频时无法获取弹幕；少数开源方案（如 Danmaku2ASS）将弹幕转换为静态字幕，丧失了滚动的动态渲染特性。

本文设计并实现了一套跨播放器通用弹幕悬浮系统。核心思想是"外挂式"架构：以 Electron 透明悬浮窗叠加于任意播放器之上，不依赖任何播放器接口；后端服务聚合 Bilibili（XML/Protobuf 双协议）、腾讯视频、芒果TV、直播吧、腾讯体育五种异构弹幕源，转化为统一数据模型；前端以自研 DanmakuEngine 弹幕渲染引擎驱动，采用基于时间戳的轨道调度算法和 requestAnimationFrame 手动动画循环，解决了 Electron 透明窗口下 CSS Web Animations 不可靠的工程难题。系统还实现了基于 mpv JSON IPC 协议的 IINA 播放器深度联动（播放进度、暂停、快进全自动同步），以及桌面端与 iPad 端的多端协同。

经实际部署测试，系统在 MacBook Pro (M1 Pro) 上以 60FPS 稳定渲染数万条弹幕，内存占用低于 15MB；支持多显示器热插拔和 macOS 全屏 Space 切换；弹幕最快加载速度（XML 模式）仅需 1.2 秒。系统已持续运行超过三个月，覆盖日常视频观看、NBA 赛事直播弹幕、YouTube 视频自动下载等实际场景。

**关键词**：弹幕渲染；Electron 透明窗口；多源数据聚合；mpv JSON IPC；轨道调度算法；requestAnimationFrame

## Abstract

Danmaku, a real-time comment system overlaid on video content, has become a defining feature of East Asian online video culture. However, mainstream video platforms lock danmaku data within their proprietary players, preventing users from accessing danmaku when watching local videos through third-party players. Existing open-source solutions either bind to specific websites or degrade danmaku to static subtitles, losing the dynamic scrolling experience.

This thesis presents a cross-player danmaku overlay system. The core innovation is a "plugin-style" architecture: an Electron transparent overlay window sits atop any video player; a modular Express.js backend aggregates danmaku from five heterogeneous sources (Bilibili XML/Protobuf dual protocol, Tencent Video, Mango TV, Zhibo8, Tencent Sports) into a unified data model; and a self-developed DanmakuEngine rendering engine uses a timestamp-based lane scheduling algorithm with manual requestAnimationFrame loops to overcome the unreliability of CSS Web Animations in transparent Electron windows. The system also implements deep integration with the IINA media player via mpv JSON IPC protocol, enabling automatic synchronization of playback progress, pause/resume, and seek operations, as well as cross-device coordination between desktop and iPad.

In deployment testing, the system renders tens of thousands of danmaku at a stable 60FPS with under 15MB memory usage on an Apple M1 Pro device. It supports hot-plugging multiple displays and macOS full-screen space transitions, and has been in continuous operation for over three months across various real-world usage scenarios.

**Keywords**: danmaku rendering; Electron transparent window; multi-source data aggregation; mpv JSON IPC; lane scheduling algorithm; requestAnimationFrame

## 第一章 绪论

### 1.1 研究背景与意义

#### 1.1.1 弹幕的起源与演化

弹幕（Danmaku，日语「弾幕」，本义为"弹幕射击"）起源于 2006 年日本视频网站 niconico 动画。用户发布的评论从右至左滚动穿过视频画面，形成类似射击游戏中弹幕的视觉效果，因此得名。2009 年，中国视频网站 Bilibili（简称 B站）引入了弹幕系统并加以改进：支持彩色弹幕、固定位置弹幕、顶部弹幕等多种模式。此后，弹幕迅速从二次元亚文化圈层扩散至整个中文互联网，腾讯视频、爱奇艺、芒果TV 等主流平台纷纷引入弹幕系统。

截至 2026 年，B站单日弹幕发送量超过 3000 万条，弹幕已从简单的文字评论演变成为包含同传字幕、科普注解、节奏互动、粉丝应援等多元功能的"视频语言层"。弹幕的交互模式——"在特定视频时刻发送评论，所有观看者在同一时刻看到同一条评论"——创造了一种独特的"虚拟共时性"，使得独自观看视频变成了群体参与的文化仪式。

#### 1.1.2 弹幕生态的封闭性问题

然而，弹幕生态在繁荣的同时存在三个结构性矛盾：

**第一，平台的封闭性**。Bilibili、腾讯视频、芒果TV 等平台各自拥有独立的弹幕数据库和弹幕传输协议，弹幕作为差异化竞争力的核心组成部分被封闭在各自的网页或客户端播放器内。B站的弹幕接口（`comment.bilibili.com` 和 `api.bilibili.com/x/v2/dm/web/seg.so`）需要特定的请求参数（cid、aid、Cookie 认证）才能访问；腾讯视频的弹幕接口（`dm.video.qq.com`）同样有 Referer 校验和 VID 精度要求。用户从这些平台下载到本地的视频文件，若使用 IINA、PotPlayer、VLC 等第三方播放器观看，完全无法获取原始的弹幕内容。

**第二，设备的割裂性**。用户可能在桌面端（macOS/Windows）使用高清播放器观看视频，同时希望在 iPad 或手机等移动设备上同步弹幕显示；体育迷可能同时在电脑上看直播分析、在 iPad 上看弹幕互动。现有弹幕系统均为平台绑定、设备绑定，缺乏统一的会话状态管理和跨设备协同机制。用户在桌面端保存的弹幕配置、播放进度、直播流地址等，无法在移动端自动恢复。

**第三，播放器的依赖性**。少数开源项目尝试将弹幕导入第三方播放器，但均存在明显局限。Danmaku2ASS 将弹幕 XML 文件预处理转换为 ASS（Advanced SubStation Alpha）字幕格式，这些静态字幕可被大多数播放器加载，但丧失了弹幕原本的实时滚动动画、轨道调度、碰撞避免等动态渲染特性。弹弹Play 是一款 Windows 平台的弹幕播放器，但其弹幕引擎绑定于特定播放器，不支持 macOS 平台，且不覆盖直播弹幕等实时场景。

#### 1.1.3 研究意义

设计一套播放器无关、多源聚合、跨设备协同的弹幕系统，具有以下意义：

从**用户体验**角度，打破了"哪个平台看视频就在哪个平台看弹幕"的限制，用户可以在任何播放器上享受弹幕文化体验，同时保持播放器本身的性能优势（如 IINA 的硬件解码、色彩管理）。

从**技术探索**角度，系统面临多个非平凡的工程挑战：如何在 Electron 透明窗口的受限渲染管道中实现高性能的弹幕动画？如何适配五种使用不同协议（XML、Protobuf、分片 JSON、游标分页、增量轮询）的异构弹幕源？如何在第三方播放器（IINA）与弹幕系统之间建立可靠的双向同步？这些问题的解决对类似系统的设计具有参考价值。

从**生态互补**角度，本系统不是要取代 B站 或其他视频平台，而是为"本地视频观看"这一平台无法覆盖的场景提供弹幕体验。利用 B站 的公开 API 获取弹幕数据属于合理使用范畴——用户已经拥有视频文件（通过合法渠道），弹幕数据本身是用户生成内容（UGC），不应被平台以技术手段封锁。

### 1.2 国内外研究现状

#### 1.2.1 浏览器扩展方案：Bilibili-Evolved

Bilibili-Evolved 是 GitHub 上 Star 数超过 20,000 的知名 B站增强浏览器扩展，使用 TypeScript 编写，提供弹幕下载、主题定制、去广告、快捷键增强等功能。其弹幕下载功能支持将视频弹幕导出为 XML 或 JSON 格式文件。

然而，Bilibili-Evolved 的核心局限在于**只能在 B站网页端运行**。它是一个典型的"增强脚本"——运行在 `bilibili.com` 域名下的 Tampermonkey/UserScript 环境中，修改和增强 B站网页播放器的行为。它无法脱离 B站网页播放器独立运行，无法将弹幕叠加到本地视频或第三方播放器上。从架构角度看，Bilibili-Evolved 的弹幕下载能力可以与本系统形成互补——用户可使用 Bilibili-Evolved 下载弹幕文件，然后在本地通过本系统加载弹幕。

#### 1.2.2 字幕转换方案：Danmaku2ASS

Danmaku2ASS 是由 m13253 维护的 Python 命令行工具，发表于 GitHub（约 1,500 Stars），专用于将 B站弹幕 XML 文件转换为 ASS（Advanced SubStation Alpha）高级字幕格式。ASS 格式支持文本定位、字体样式、淡入淡出等基本特效，可被大多数桌面播放器加载。

Danmaku2ASS 的工作原理是：读取 B站弹幕 XML → 解析每条弹幕的时间、位置、颜色、模式 → 生成 ASS 字幕文件（定义每条弹幕为 ASS 事件） → 播放器加载字幕文件。

但该方案存在四个根本性缺陷：（1）弹幕被"降级"为字幕——丧失了弹幕引擎的实时轨道调度能力，所有弹幕只能静态显示在固定的 ASS 层上，无法实现碰撞避免和动态轨道分配；（2）预处理模式——每次更换视频都需要执行一次转换流程（`python danmaku2ass.py input.xml -o output.ass`），不支持在线实时加载；（3）播放器兼容性参差不齐——不同播放器对 ASS 标签的支持程度各不相同，LibreOffice/ffmpeg 系列的播放器（如 mpv/IINA）对 ASS 支持较好，但 QuickTime、Windows Media Player 等对 ASS 支持极差；（4）仅支持 B站弹幕——无法处理腾讯视频、芒果TV 等其他平台的弹幕格式。

#### 1.2.3 集成播放器方案：弹弹Play

弹弹Play 是一款 Windows 平台的专用弹幕播放器，具有"弹幕池"功能——用户输入 BV 号后，播放器自动从 B站下载弹幕并叠加在视频上。它是目前 Windows 平台上最成熟的弹幕播放方案之一。

其局限性包括：（1）绑定特定播放器——用户必须使用弹弹Play 内置的播放组件，无法使用 IINA、PotPlayer 等偏好播放器；（2）仅支持 Windows 平台，不支持 macOS 或移动端；（3）不支持直播弹幕场景（如 NBA 赛事直播的文字弹幕）；（4）弹幕源单一，仅支持 B站。

### 1.3 研究内容

本文的研究目标是设计并实现一套多源弹幕聚合与跨播放器透明悬浮系统，具体研究内容包括：

**1. 多源弹幕数据聚合与统一建模**。研究 Bilibili（XML 和 Protobuf 双协议）、腾讯视频（分片 JSON API）、芒果TV（索引探测式 JSON API）、直播吧（分页 JSON API）、腾讯体育（POST + cursor 分页 API）五种弹幕源的接口特征和数据格式，设计统一的弹幕数据模型，将异构数据源归一化为标准化的 `{text, time, color, mode, size, ctime}` 六元组结构。

**2. 透明窗口下的弹幕渲染引擎设计**。研究 Electron 透明窗口的渲染特性，分析 CSS Web Animations API 在透明合成管道中的失败机制，设计基于轨道调度算法和 requestAnimationFrame 手动动画循环的 DanmakuEngine 弹幕渲染引擎，实现 DOM 对象池复用和 GPU 加速的 translate3d 变换优化。

**3. 播放器间进程的实时通信与同步**。研究 mpv JSON IPC 协议，设计 Python 轮询脚本通过 Unix Domain Socket 与 IINA 播放器进行双向通信，实现播放进度、暂停/播放状态、快进/快退操作的实时同步，以及视频文件切换的自动弹幕源识别与加载。

**4. 跨设备会话协同机制**。研究基于服务端持久化的跨设备状态同步方案，设计"iPad 设置"数据结构和增量保存策略，实现桌面端与移动端的弹幕配置、直播流地址、播放进度的无缝漫游。

**5. 系统安全与鲁棒性设计**。研究文件路径穿越防护、Cookie 安全传输、僵尸 Socket 检测与恢复、视频过期自动清理等工程安全与可靠性机制。

## 第二章 相关技术基础

### 2.1 Electron 桌面应用框架

#### 2.1.1 框架概述

Electron 是 GitHub 于 2013 年开源的跨平台桌面应用框架，使用 Chromium 作为前端渲染引擎、Node.js 作为后端运行时，允许开发者使用 Web 技术（HTML、CSS、JavaScript）构建桌面应用程序。知名产品包括 Visual Studio Code、Slack、Discord、Figma 等均基于 Electron 构建。

Electron 应用的架构分为两个进程：主进程（Main Process）负责管理窗口生命周期、系统级 API 调用、全局快捷键注册等；渲染进程（Renderer Process）运行在 Chromium 沙盒中，负责页面渲染和用户交互。两个进程之间通过 IPC（Inter-Process Communication）通信。

#### 2.1.2 透明窗口机制

Electron 对本系统的核心价值在于其透明窗口能力。通过 BrowserWindow 构造时的配置参数组合，可以创建一个完全透明、无边框、始终置顶的窗口：

```javascript
const win = new BrowserWindow({
  transparent: true,          // 启用透明渲染管道（WebGL 合成）
  frame: false,               // 隐藏窗口边框和标题栏
  alwaysOnTop: true,          // 窗口始终位于最顶层
  skipTaskbar: true,          // 不在任务栏显示
  resizable: false,           // 禁止窗口缩放
  hasShadow: false,           // 禁用窗口阴影（防止阴影穿透）
  backgroundColor: '#00000000' // 背景色设为完全透明（ARGB）
});
```

其中，`transparent: true` 是 Electron 透明窗口的核心开关。启用后，Chromium 使用 WebGL 渲染管道而非标准的合成管道，页面的 `background-color: transparent` 区域在屏幕显示为完全透明，下层窗口和应用的内容直接透过这些区域可见。

`setIgnoreMouseEvents(true, { forward: true })` 是另一关键配置：将悬浮窗设为鼠标事件穿透模式，用户在弹幕层上的所有点击、拖拽、滚轮操作都会被转发到悬浮窗下方的窗口（即被覆盖的播放器），使得弹幕悬浮窗对播放器的操作"完全隐形"。当用户唤起控制面板时，调用 `setIgnoreMouseEvents(false)` 临时关闭穿透，以便面板上的按钮可以交互。

#### 2.1.3 全局快捷键

Electron 的 `globalShortcut` 模块允许注册操作系统级别的组合键监听，即使应用窗口没有获得焦点也能响应。本系统注册了 11 个全局快捷键（不含 `Esc` 键，避免与播放器原生 Esc 全屏退出冲突），包括 `Cmd+Space`（播放/暂停）、`Cmd+←/→`（快退/快进 0.5s）、`Cmd+Shift+←/→`（快退/快进 2s）、`Cmd+↑/↓`（透明度 ±5%）、`Cmd+Shift+↑/↓`（字号 ±2px）、`Cmd+O`（显示/隐藏面板）、`Cmd+L`（加载弹幕）、`Cmd+Q`（退出）。

快捷键触发后的操作通过 `win.webContents.send('shortcut:xxx')` 传递到渲染进程，经过 `preload.js` 的 `contextBridge` 白名单校验后到达前端处理。

#### 2.1.4 多显示器与 Space 兼容

macOS 全屏 Space 切换机制会导致 Electron `alwaysOnTop` 窗口在某些情况下丢失置顶状态。本系统实现了 3 秒间隔的"可见性心跳"（Visibility Heartbeat）机制：定时器每 3 秒重新执行 `win.setAlwaysOnTop(true, 'screen-saver')` 和 `win.setVisibleOnAllWorkspaces(true, { visibleOnFullScreen: true })`，强制确保悬浮窗在任何 Space 中都保持可见。

多显示器支持通过 `screen.getAllDisplays()` 获取所有显示器信息，为每个显示器创建独立的覆盖窗口。监听 `display-added` 和 `display-removed` 事件，实现热插拔显示器的动态适配。移除显示器时通过窗口 bounds 位置（x, y 坐标）匹配对应的覆盖窗口并关闭。

### 2.2 mpv 与 JSON IPC 协议

#### 2.2.1 mpv 播放器

mpv 是一款功能强大的开源跨平台媒体播放器，fork 自 MPlayer 和 mplayer2 项目，以其高性能、高可定制性和丰富的脚本接口而著称。IINA 是 macOS 平台上的 mpv 前端，由独立开发者 Collider LI 维护，被广泛认为是"macOS 上最好的现代播放器"，在 App Store 上获得 4.8/5.0 的评分。

#### 2.2.2 JSON IPC 通信协议

mpv 在编译时支持通过 `--input-ipc-server` 选项开启 JSON IPC 服务。配置 `input-ipc-server=/tmp/iina-socket` 后，mpv 在启动时会创建一个 Unix Domain Socket 文件，外部程序可通过标准的 Socket 通信方式（`AF_UNIX, SOCK_STREAM`）向 mpv 发送 JSON 格式的命令并接收 JSON 格式的响应。

每个 JSON 命令是一个包含 `command` 数组的对象，如：

```json
{"command": ["get_property", "time-pos"]}
```

响应格式为：

```json
{"data": 123.456, "request_id": 0, "error": "success"}
```

本系统使用以下关键 API：
- `get_property("path")`：获取当前播放文件的绝对路径
- `get_property("time-pos")`：获取当前播放位置（浮点秒数）
- `get_property("pause")`：查询暂停状态（布尔值）
- `set_property("pause", bool)`：设置暂停/开始播放

2 秒的超时设置（`socket.settimeout(2)`）确保在 mpv 无响应时不会无限阻塞。通信完成后主动关闭 socket 连接（`sock.close()`），避免占用系统文件描述符。

#### 2.2.3 Unix Domain Socket 的可靠性问题

在实际使用中发现，当 IINA 进程异常退出（如通过 Activity Monitor 强制结束、系统崩溃）时，`/tmp/iina-socket` 文件可能残留在文件系统中，但其对端已无进程监听，成为"僵尸 Socket"。此时 `os.path.exists(SOCKET_PATH)` 返回 `True`，但 `socket.connect()` 会抛出异常。

本系统设计了连续失败计数器（`fail_count`）和阈值（`MAX_FAIL = 5`）来处理此问题。连续 5 次（每次轮询间隔 3 秒，共 15 秒）连接失败后，判定 Socket 为僵尸状态，重置所有内部状态变量（`last_path = ""`, `last_pos = 0`, `fail_count = 0`），提示"等待 IINA 重启"。

#### 2.2.4 Python 轮询脚本设计

轮询脚本（`iina-watcher.py`）的轮询间隔选择 3 秒，是在"同步及时性"与"系统资源占用"之间的折中。3 秒的间隔意味着最坏情况下的时间同步误差为 3 秒（弹幕相对于视频可能偏移 3 秒），对于以分钟为单位的视频观看体验而言，这一误差在可接受范围内。若缩短为 1 秒，同步精度提升有限（1.5 秒 → 0.5 秒），但 Socket 连接频率增加 3 倍，在多任务桌面环境下的综合开销不可忽略。

### 2.3 Protocol Buffers

#### 2.3.1 技术概述

Protocol Buffers（简称 Protobuf）是 Google 开发的一种语言无关、平台无关的二进制序列化格式，于 2008 年开源。与 XML 和 JSON 相比，Protobuf 编码后的数据体积更小（通常为 XML 的 1/3 至 1/10）、解析速度更快（通常为 XML 的 20-100 倍）。这些特性使其非常适合网络传输场景，特别是移动网络和实时通信场景。

#### 2.3.2 B站弹幕的 Protobuf 应用

Bilibili 的新版弹幕接口（称为"seg.so"接口）使用 Protobuf 编码返回弹幕数据。接口将视频按 360 秒（6 分钟）切分为分段（segment），每个分段独立返回一条 Protobuf 编码的 `DmSegMobileReply` 消息。

本系统使用 `protobufjs` 库在 Node.js 端进行解码。首先在代码中嵌入 `.proto` 定义：

```protobuf
message DanmakuElem {
  int64 id = 1;
  int32 progress = 2;
  int32 mode = 3;
  int32 fontsize = 4;
  uint32 color = 5;
  string midHash = 6;
  string content = 7;
  int64 ctime = 8;
  int32 weight = 9;
  string action = 10;
  int32 pool = 11;
  string idStr = 12;
  int32 attr = 13;
}

message DmSegMobileReply {
  repeated DanmakuElem elems = 1;
}
```

利用 `protobuf.parse()` 编译 `.proto` 文件为 JavaScript 类型定义，然后通过 `DmSegMobileReply.decode(buffer)` 解码二进制数据。注意关键字段的转换：`progress` 是毫秒时间戳，需除以 1000 转换为秒（`Number(item.progress) / 1000`）；`color` 是 32 位无符号整数颜色值，需转换为 CSS 兼容的 `#RRGGBB` 十六进制格式（`'#' + Number(item.color).toString(16).padStart(6, '0')`）。

Protobuf 的二进制特性要求 HTTP 请求设置 `responseType: 'arraybuffer'`，否则 axios 会将响应自动解析为 UTF-8 字符串，导致二进制数据损坏。请求完成后，使用 `Buffer.from(res.data)` 将 ArrayBuffer 包装为 Node.js Buffer 对象传入解码器。

### 2.4 ffmpeg 与硬件加速转码

#### 2.4.1 核心功能

ffmpeg 是业界标准的开源音视频处理工具集，提供编解码、格式转换、流处理等功能。本系统利用 ffmpeg 实现三个核心能力：视频实时转码、缩略图生成和视频探测（ffprobe）。

#### 2.4.2 硬件加速方案

在 macOS 平台上，利用 Apple 的 VideoToolbox 框架实现视频编码的硬件加速。`h264_videotoolbox` 编码器调用 Mac 芯片内置的 H.264 硬件编码单元，相比软件编码器（如 `libx264`），CPU 占用降低 70-80%，编码速度提升 3-5 倍。

转码管线如下：`-hwaccel videotoolbox` 开启硬件解码加速 → `-i videoPath` 读取源文件 → `-c:v h264_videotoolbox` 指定 H.264 硬件编码器 → `-b:v 6000k` 设置视频码率 → `-color_primaries bt709 -color_trc bt709 -colorspace bt709` 设置色彩空间 → `-pix_fmt yuv420p` 设置像素格式（最大兼容性）→ `-c:a aac -b:a 128k` 音频重编码为 AAC → `-movflags frag_keyframe+empty_moov+default_base_moof` 生成可分片传输的 MP4 片段 → `pipe:1` 输出到 stdout。

`-movflags frag_keyframe+empty_moov+default_base_moof` 三个标志的组合是实时转码流式输出的关键：`frag_keyframe` 在每个关键帧后创建一个 MP4 片段；`empty_moov` 不在文件头部写入 moov atom（通常位于文件末尾，实时流无法预知）；`default_base_moof` 使用默认的 moof 基准。三者结合使得客户端可以在转码过程中边下载边播放，无需等待整个文件转码完毕。

#### 2.4.3 缩略图生成

使用 ffmpeg 的 `-ss`（seek）和 `-frames:v`（帧数限制）参数快速截取视频缩略图：

```bash
ffmpeg -y -ss 00:00:05 -i video.mp4 -frames:v 1 -q:v 2 thumb.jpg
```

`-ss 00:00:05` 定位到第 5 秒位置（避免视频开头的黑色帧或片头 Logo），`-frames:v 1` 只输出一帧图像，`-q:v 2` 设置 JPEG 输出质量（1=最佳，31=最差）。这张缩略图缓存到 `cache/thumbs/` 目录，文件名为视频文件名 + `.jpg` 后缀（如 `EP01_机智的医生生活_ep703938.mp4.jpg`）。服务端的 `getThumbPath()` 函数通过 `path.basename(fileName) + '.jpg'` 生成统一命名，确保缩略图的可发现性。

#### 2.4.4 ffprobe 视频元信息提取

转码任务启动前，使用 ffprobe（ffmpeg 套件中的分析工具）获取视频总时长：

```bash
ffprobe -v error -show_entries format=duration \
  -of default=noprint_wrappers=1:nokey=1 video.mp4
```

输出为浮点秒数。该时长信息用于前端进度条的范围设定和转码进度的百分比计算。ffprobe 的 `-v error` 参数仅输出错误级别信息，避免干扰 duration 值的提取。

### 2.5 前端关键技术

#### 2.5.1 HLS 流媒体协议

HTTP Live Streaming（HLS）是 Apple 于 2009 年提出的自适应比特率流媒体传输协议。一个 HLS 流由 `.m3u8` 播放列表文件和一系列的 `.ts` 媒体分片文件组成。播放列表可以是主列表（引用多个不同码率的子列表，客户端自适应选择）或媒体列表（直接引用媒体分片）。

本系统的 iPad 端支持加载外部 HLS `.m3u8` 直播流地址。对于原生支持 HLS 的浏览器（Safari、iOS Safari），直接设置 `<video src=".m3u8">` 即可播放；对于其他浏览器（Chrome、Edge），使用 hls.js 库（JavaScript HLS 客户端）进行软解码播放。HLS 的分片架构天然适应移动网络：每个 `.ts` 分片通常为 2-10 秒的视频片段，客户端可以根据当前网络状况动态选择不同码率的分片，在直播场景中实现无缓冲的流畅播放。

#### 2.5.2 PWA 与 iPad 全屏体验

Progressive Web Application（PWA）是一组 Web 技术标准的集合，旨在使 Web 应用获得接近原生应用的体验。本系统为 iPad 端配置了以下 PWA 特性：

- `meta[name="apple-mobile-web-app-capable"][content="yes"]`：允许页面作为独立 Web 应用运行（隐藏 Safari 地址栏和工具栏）
- `meta[name="apple-mobile-web-app-title"][content="弹幕直播"]`：添加到主屏幕后的应用标题
- `link[rel="apple-touch-icon"]`：主屏幕图标（粉色背景，适配 iPad 分辨率）
- `meta[name="apple-mobile-web-app-status-bar-style"][content="black-translucent"]`：状态栏样式
- `meta[name="viewport"][content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no"]`：锁定缩放，防止双击放大干扰弹幕触控

#### 2.5.3 requestAnimationFrame 与 CSS Animations 的取舍

在正常的浏览器渲染管道中，CSS Animations 和 Web Animations API 是推荐的高性能动画方案——浏览器可以在合成器线程上独立处理这些动画，不阻塞主线程的 JavaScript 执行。然而，在 Electron 的透明窗口模式下，由于透明合成使用了特殊的 WebGL 渲染管道而非标准的 GPU 合成器，CSS Animations 行为变得不可靠。实际测试发现：CSS `@keyframes` 在透明窗口中会出现随机掉帧、动画暂停、帧率降至个位数的问题。

`requestAnimationFrame`（rAF）回调在主线程的渲染帧之间执行，可以手动控制每一步的 DOM 属性更新。虽然 rAF 本质上是 CPU 驱动的（在主线程上执行 JavaScript），但由于本系统的弹幕渲染仅需要更新元素的 `transform` 属性（GPU 加速的合成属性，不触发 layout/paint），实际性能开销极低。在万条弹幕的压测中，rAF 回调的单帧执行时间稳定在 2-5ms，远低于 16.67ms（60FPS）的帧预算。

## 第三章 系统设计

### 3.1 设计目标与原则

#### 3.1.1 设计目标

本系统的设计围绕四个核心目标展开：

**G1：播放器无关性**。弹幕悬浮窗不依赖任何特定播放器的接口，作为独立的系统级透明窗口运行，可叠加于 IINA、PotPlayer、VLC、浏览器等任意播放器之上。

**G2：多源弹幕兼容**。支持 B站（XML + Protobuf）、腾讯视频、芒果TV、直播吧、腾讯体育五种弹幕源，覆盖 VOD 点播（一次性全量加载）和直播（增量轮询加载）两大类场景。

**G3：播放器联动**。对于支持 JSON IPC 的 IINA 播放器，实现播放进度、暂停/播放、快进/快退的自动同步；对于不支持 IPC 的播放器，提供手动时间轴同步（通过滑块、快捷键调整弹幕偏移）。

**G4：跨设备协同**。桌面端和移动端共享弹幕配置、直播流地址、播放进度等状态，任一设备更新后其他设备可立即同步。

#### 3.1.2 设计原则

- **模块化**：后端按功能域拆分为独立的路由模块（`routes/`）、服务模块（`services/`）和共享模块（`shared/`），前端弹幕引擎（`danmaku-core.js`）与页面逻辑（`overlay.js`/`ipad.js`）解耦。
- **渐进增强**：核心弹幕加载功能可在任意浏览器中运行，无需 Electron；iPad 端为零依赖的 HTML/JS 页面，仅需浏览器。
- **防御性设计**：在系统的每个边界（用户输入、文件路径、外部 API 响应、网络连接）实施安全校验和容错处理，确保单点故障不影响整体可用性。
- **离线优先**：弹幕数据和视频元信息均支持缓存，在无网络或 API 不可达时仍然可用（使用已缓存的弹幕）。

### 3.2 总体架构

本系统采用前后端分离的三层架构：

**第一层：Electron 桌面壳层**。负责创建透明悬浮窗、注册全局快捷键、管理多显示器布局、提供 IPC 安全桥接。打包模式（`electron/app.js`）下内嵌 Express 服务器，用户无需手动启动后端。

**第二层：Express API 服务层**。提供 RESTful API 接口，包括弹幕数据获取（`/api/danmaku`）、视频流传输（`/stream`、`/video/:name`）、文件夹管理（`/api/folders`）、播放进度存取（`/api/progress`）、IINA 状态同步（`/api/iina-state`）、转码任务管理（`/api/convert`）、配置管理（`/api/overlay-config`、`/api/ipad-settings`）等。服务层 (`services/`) 封装各弹幕源的 API 调用和数据转换逻辑。路由模块按功能域拆分为七个独立的 Express Router。

**第三层：前端渲染层**。包含三个 HTML 页面：`video.html`（Web 播放器，含视频播放和弹幕设置）、`overlay.html`（Electron 透明悬浮窗，覆盖于任意播放器之上）、`ipad.html`（iPad 触屏优化版，支持 HLS 直播流+实时弹幕+触控面板+PWA 主屏幕图标）。所有页面的弹幕渲染由共享的 `DanmakuEngine`（`danmaku-core.js`）驱动。

### 3.3 模块划分与接口设计

#### 3.3.1 后端路由模块

系统将 1222 行的原始单体 `server.js` 重构为七个独立的路由模块，每个模块负责一组逻辑内聚的端点：

| 路由文件 | 挂载路径 | 核心端点 | 职责 |
|---------|---------|---------|------|
| danmaku.js | `/api` | GET /danmaku, GET /danmu, GET /danmu/progress | 五源弹幕聚合、缓存管理、加载进度推送 |
| video.js | `/api` | GET /videos, GET /thumbnail, PUT /config | 视频列表、缩略图、视频目录配置 |
| library.js | `/api` | GET /folders, GET /video-files, POST /rename | 文件夹扫描、B站番剧批量重命名 |
| progress.js | `/api` | GET/POST /progress, GET/POST /iina-state | 播放进度存取、IINA 状态同步 |
| convert.js | `/api` | POST /convert, GET /convert/status, GET /convert/history | ffmpeg 转码任务生命周期管理 |

| config.js | `/api` | GET/PUT /overlay-config, GET/PUT /folder-history, GET/PUT /ipad-settings | 弹幕配置、文件夹历史、iPad 跨设备设置 |

#### 3.3.2 共享模块设计

为解决路由拆分后的跨模块状态共享问题，设计了 `shared/` 共享模块：

- **constants.js**：导出所有缓存目录路径、配置文件路径和环境变量派生值（`CACHE_DIR`、`PLAYBACK_DIR`、`DANMU_DIR`、`THUMB_DIR`、`META_DIR`、`FOLDERS_BASE`、`RETENTION_CONFIG_FILE` 等），所有路由模块引用同一份常量定义，确保路径一致性。
- **state.js**：导出一个可变对象 `{ videoDir, danmuProgressMap, convertTasks, iinaState }`。所有路由模块共享此对象的同一引用，对 `state.videoDir` 的修改在所有模块间立即可见，解决了 Node.js 模块缓存机制下原始类型值无法跨模块共享的问题。
- **helpers.js**：提供跨模块使用的公共函数，包括路径安全校验（`resolveLibraryDirectory`、`resolveLibraryVideoFile`）、目录扫描（`scanFolders`、`scanVideoFiles`）、转换历史持久化（`saveConvertHistory`）和错误类型判断（`isPathValidationError`）。

### 3.4 统一弹幕数据模型

不同弹幕源的原始数据格式差异巨大。为在前端实现统一的渲染管线，系统定义了一个六元组弹幕数据模型作为中间表示：

```javascript
// 标准化弹幕对象
{
  text:  "前面的区域以后再来探索吧？",  // 弹幕文本内容
  time:  34.567,                        // 视频时间偏移（秒，浮点数）
  color: "#66ccff",                     // 文字颜色（CSS 十六进制格式）
  mode:  1,                             // 显示模式: 1=滚动, 4=底部, 5=顶部
  size:  25,                            // 字号（像素值）
  ctime: 1712345678                     // 发送时间（Unix 秒时间戳，可选）
}
```

各弹幕源的字段映射关系如下：

**B站 XML**：`<d p="time,mode,size,color,ctime,pool,sender,dbId">text</d>`，通过 `p` 属性的逗号分隔字段获取时间、模式等参数。

**B站 Protobuf**：`DanmakuElem.progress(ms) → time(s)`、`DanmakuElem.mode → mode`、`DanmakuElem.fontsize → size`、`DanmakuElem.color(uint32) → #RRGGBB`、`DanmakuElem.content → text`、`DanmakuElem.ctime → ctime`。

**腾讯视频**：`barrage_list[].time_offset(ms) → time(s)`、`content_style.gradient_colors[0] → color`、`barrage_list[].content → text`、`create_time → ctime`，mode 固定为 `scroll`。

**芒果TV**：`items[].time(ms) → time(s)`、`items[].content → text`，color 固定为 `#ffffff`，mode 固定为 `scroll`。

**直播吧**：`item.createtime → rawTime`、`item.content → text`、`item.room == 2 ? #ffcc00 : #ffffff → color`，mode 固定为 1，time 固定为 0（由前端填充当前播放时间）。

**腾讯体育**：`danmuku_info.content.content_v2 → text`、`danmaku_control_info.post_time → ctime`，color 固定为 `#ffffff`，mode 固定为 1，time 固定为 0（由前端填充当前播放时间）。

### 3.5 弹幕缓存系统设计

对于 VOD 点播类弹幕源（B站、腾讯视频、芒果TV），弹幕数据在视频发布后即固定不变。对这些弹幕源启用缓存可以显著提升用户体验（即时加载、离线可用）并降低对第三方 API 的请求频率。

**缓存策略**：首次加载弹幕时，将标准化后的弹幕数组（连同 source、id、cid、count 等元信息）序列化为 JSON 文件存入 `cache/danmu/` 目录。文件命名规则为 `{videoId}.{strategy}.json`，其中 strategy 为 `seg_so` 或 `xml`（B站）或 `qq`（腾讯）或 `mango_{timeStr}`（芒果）。后续请求命中缓存时，`/api/danmaku` 直接返回缓存 JSON，并标记 `fromCache: true`。

**缓存刷新**：前端传递 `refresh=1` 参数时，跳过缓存，向第三方 API 发起新请求。刷新前，旧缓存文件自动备份为 `{filename}.bak.YYYYMMDDHHmm` 格式（时间戳精确到分钟）。备份机制的设计考虑是：如果新拉取的弹幕数据异常（如 API 暂时不可用返回空数据），用户可以通过备份文件手动恢复。

**直播源不缓存**：直播吧和腾讯体育的弹幕是实时的（每秒新增），缓存无意义。每次请求都实时从第三方 API 拉取。

### 3.6 跨设备同步架构

传统弹幕系统的一个核心局限是**设备绑定**——桌面端保存的设置和进度无法在移动端同步。本系统设计了基于服务端持久化的跨设备同步架构。

**iPad 设置同步（`cache/ipad_settings.json`）**：保存以下完整会话状态：

```json
{
  "streamUrl": "http://xxx/live.m3u8",
  "matchId": "12345",
  "txspRoomId": "1532819195",
  "txspProgramId": "366831",
  "txspCookie": "qq_domain_video_guid_verify=xxx; ...",
  "txspLastSeq": 12345,
  "txspCursor": "xxx",
  "source": "txsp"
}
```

前端通过 500ms 防抖（`debounce`）机制在用户修改设置后自动保存到服务端。debounce 延迟避免了键盘输入的每个字符都触发一次 HTTP 请求。除了输入框的 `input` 事件触发保存外，粘贴提取（bookmarklet JSON 粘贴自动填充）和加载弹幕（`loadDanmaku` 调用）也会触发立即保存（绕过 debounce），确保关键状态不丢失。

**播放进度同步（`playback/` 目录）**：每个视频文件对应一个 JSON 文件，存储最近播放时间戳：

```json
{
  "id": "/full/path/to/video.mp4",
  "time": 1234,
  "updatedAt": 1712345678000
}
```

IINA 联动通过 `iina-watcher.py` 每隔 3 秒检测 IINA 播放位置变化时调用 `POST /api/progress` 保存进度。iPad 端和桌面 Overlay 端在选中视频文件后调用 `GET /api/progress` 恢复进度。文件命名使用全角斜杠字符"／"（Unicode FF0F）替换 Unix 路径分隔符"/"，以兼容文件名不能包含"/"的文件系统限制。

## 第四章 系统实现

### 4.1 DanmakuEngine 弹幕渲染引擎

DanmakuEngine 是系统的前端核心，是一个约 330 行的纯 JavaScript 类（ES5 兼容语法，支持 module.exports 导出），不依赖任何第三方框架或库，只需要一个 DOM 容器元素即可工作。桌面端（`overlay.js`）和 iPad 端（`ipad.js`）共享同一份引擎代码。

#### 4.1.1 轨道调度算法

弹幕渲染的核心难题是**避免多条弹幕在同一时间发生水平重叠**，同时**最大化屏幕空间的利用效率**。

DanmakuEngine 采用基于时间戳的轨道（Lane）调度算法。算法将屏幕可视区域的纵向空间划分为 N 个等高轨道。轨道高度由当前字号加间距决定：`laneHeight = fontSize + 8px`。轨道数量由弹幕显示区域占比（`area` 参数，默认 25% 即屏幕上半部分）动态计算：`laneCount = max(1, floor(availableHeight / laneHeight))`。

每个轨道维护一个时间戳 `_laneAvailableAt[i]`，表示该轨道"从何时起可用"。当一条新弹幕准备发射时，算法按以下步骤分配轨道：

1. 遍历所有轨道，查找第一个 `_laneAvailableAt[i] <= performance.now()`（当前时间）的可直接使用轨道。
2. 若所有轨道均被占用，选择 `_laneAvailableAt[]` 最小值对应的轨道（最早被释放的轨道），弹幕在其释放后立即发射。
3. 根据弹幕宽度（通过 Canvas 2D `measureText()` 预先测量）、屏幕宽度和滚动速度计算该轨道的占用时长：`waitSeconds = (danmuWidth + laneGap) / (totalDistance / speed)`，其中 `totalDistance = width + danmuWidth`（弹幕从左边缘完全进入屏幕到右边缘完全离开的总行程），`laneGap = 24px` 是相邻弹幕间的最小间距。
4. 更新轨道时间戳：`_laneAvailableAt[laneIndex] = performance.now() + waitSeconds * 1000`。

该算法的时间复杂度为 O(N)（N 为轨道数，通常 5-20），在每帧最多发射 6 条弹幕的限制下，单帧的轨道搜索开销可忽略不计（< 0.1ms）。

#### 4.1.2 DOM 对象池

在弹幕高密度场景下（如片头片尾、剧情高潮段），每帧可能需要同时渲染 20-30 条活跃弹幕。频繁的 `document.createElement()` 和 `element.remove()` 会引发浏览器的大量重排（reflow）和重绘（repaint），对弹幕这种每帧都需要更新 DOM 属性的高频渲染场景造成明显的性能退化。

DanmakuEngine 实现了基于数组栈的 DOM 对象池：

```javascript
// 获取元素：优先从池中复用
_getDom() {
  if (this._pool.length > 0) return this._pool.pop();
  // 池空则创建新元素（预设高性能 CSS 属性）
  const el = document.createElement('div');
  el.style.contain = 'layout style paint';  // 隔离布局和绘制
  el.style.willChange = 'transform';         // 提示 GPU 预分配合成层
  el.style.backfaceVisibility = 'hidden';    // 隐藏元素背面
  el.style.pointerEvents = 'none';           // 鼠标穿透
  this.container.appendChild(el);
  return el;
}

// 回收元素：移到屏幕外后回池（不删除 DOM 节点）
_tickActive(now) {
  for (let i = this._active.length - 1; i >= 0; i--) {
    const a = this._active[i];
    const elapsed = now - a.startMs;
    if (elapsed >= a.durationMs) {
      a.el.style.transform = 'translate3d(-9999px, -9999px, 0)'; // 移出视口
      this._pool.push(a.el);  // 回池
      this._active.splice(i, 1);  // 从活跃列表移除
    }
  }
}
```

关键的设计注意事项是"移出视口再回池"的顺序：先将 DOM 元素移动到 `(-9999px, -9999px)` 的不可见位置，再将其放入对象池。如果先回池再移动，复用时元素可能短暂出现在屏幕上之前的位置，造成"闪烁"（flicker）的视觉瑕疵。`contain: layout style paint` 这条 CSS 属性告知浏览器该元素的布局/样式/绘制是独立的，对元素属性的修改不会触发父元素或兄弟元素的重排。

#### 4.1.3 requestAnimationFrame 手动动画循环

在标准浏览器环境中，CSS `@keyframes` 动画和 Web Animations API 是推荐的动画方案——它们运行在合成器线程上，独立于主线程的 JavaScript 执行，可以保证流畅的帧率。然而，在 Electron 的透明窗口模式下，由于透明合成所需的 WebGL 渲染管道与标准 GPU 合成器存在兼容性问题，CSS Animations 行为不可靠。

DanmakuEngine 采用 `requestAnimationFrame`（rAF）实现完全手动的动画循环：

```javascript
_loop() {
  if (!this._running || this._paused) return;
  const tick = (now) => {
    if (!this._running || this._paused) { this._loopId = 0; return; }
    this._tickActive(now);  // 更新活跃弹幕的 translate3d 位置
    const dt = (now - this._lastPerfTime) / 1000;  // 自上次帧的时间增量
    const currentTime = this._lastVideoTime + dt;   // 插值当前视频时间
    this._flush(currentTime);  // 按时间戳发射新弹幕
    this._loopId = requestAnimationFrame(tick);  // 注册下一帧回调
  };
  this._loopId = requestAnimationFrame(tick);
}
```

手动动画循环的核心是对时间的精确控制。系统维护两个时间基准：`_lastVideoTime` 记录上一次 seek 或 start 时的视频播放时间，`_lastPerfTime` 记录那个时刻的 `performance.now()` 值。当前帧的视频时间通过线性插值计算：`currentTime = _lastVideoTime + (performance.now() - _lastPerfTime) / 1000`。这种设计使得弹幕的时间轴独立于任何外部时钟，在播放、暂停、seek 操作之间平滑过渡。

#### 4.1.4 弹幕发射控制

`_flush()` 方法负责按时间戳发射新弹幕：遍历已排序的弹幕数组 `this.danmus[]`，从当前索引 `this.danmuIndex` 开始，找出所有 `time <= currentTime` 的弹幕并尝试发射。每帧限制最多发射 `_maxPerFrame = 6` 条弹幕，防止瞬间大量弹幕涌入造成帧率骤降。

对于超出 `_maxPerFrame` 的弹幕（即同帧第 7 条及之后的弹幕），直接跳过（`this.danmuIndex++`）而不尝试发射。这不是丢弃——弹幕本来的 `time` 就是在当前视频时刻前后的，跳过意味着它们"稍晚"一点显示，在密集弹幕场景中这种微小的调度延迟在视觉上几乎不可察觉。

#### 4.1.5 文本宽度测量优化

弹幕宽度测量是轨道调度中频率最高的操作——每条弹幕发射前都需要测量其渲染宽度以计算轨道占用时长。直接创建 DOM 元素并测量其 `offsetWidth` 会触发浏览器的布局计算，在高频调用下性能不佳。

DanmakuEngine 使用离屏 Canvas 2D 进行文本宽度预测量：

```javascript
_measureWidth(text, fontSize) {
  const key = `${fontSize}:${text}`;
  if (this._widthCache.has(key)) return this._widthCache.get(key);
  this._measureCtx.font = `bold ${fontSize}px "PingFang SC", "Microsoft YaHei", sans-serif`;
  const w = Math.ceil(this._measureCtx.measureText(text).width);
  this._widthCache.set(key, w);
  return w;
}
```

Canvas 2D 的 `measureText()` 在离屏 Canvas 上执行，不触发 DOM 布局（layout），性能远优于 DOM 测量。缓存（`_widthCache` Map）进一步避免了对相同文本的重复测量。缓存 key 包含字号（`fontSize`）以应对字号动态调整的场景——当用户通过滑块改变字号时，`configure()` 会清空整个缓存（`this._widthCache.clear()`），确保后续测量使用新的字号。

#### 4.1.6 时间戳显示功能

DanmakuEngine 支持可选的弹幕时间戳显示模式。开启 `showTimestamps: true` 后，每条弹幕前追加 `[MM:SS | YY-MM-DD HH:mm]` 格式的时间标签：

```
[12:34 | 25-04-15 20:30] 前面的区域以后再来探索吧？
```

时间标签的开销是弹幕文本变长，导致 `_measureWidth()` 返回值增大，轨道占用时间相应增加。这在弹幕密度较低的 VOD 场景中影响可忽略，但在弹幕高度密集的场景中可能导致轨道更快被占满。因此时间戳模式设计为默认关闭，由用户按需开启。

### 4.2 多源弹幕适配

#### 4.2.1 B站弹幕——双协议协调

B站弹幕的获取是五种弹幕源中实现最复杂的，涉及两个协议的协调和分段下载管理。

**视频元信息获取**：系统支持 BV 号和 EP 号两种输入。BV 号是 B站 UGC 视频的唯一标识（格式为 `BV` + 10位 Base58 编码字符），通过 `api.bilibili.com/x/web-interface/view` 接口获取元信息；EP 号是 B站番剧/影视分集的标识（`ep` + 数字），通过 `api.bilibili.com/pgc/view/web/season` 接口获取。系统通过 `id.toUpperCase().startsWith('BV')` 自动分流。元信息（aid、cid、duration、title、cover 等）缓存到 `cache/meta/` 目录，文件名格式为 `{videoId}.json`。

**XML 协议（旧版）**：接口 `comment.bilibili.com/{cid}.xml` 返回标准的 XML 文档，格式为：

```xml
<i>
  <d p="34.567,1,25,16777215,1712345678,0,abc123,1234567890">弹幕文本</d>
  <d p="35.123,4,20,16711680,1712345800,0,def456,1234567891">底部弹幕</d>
</i>
```

使用 `fast-xml-parser` 库配置 `ignoreAttributes: false` 以保留 `<d>` 标签的属性，提取 `@_p` 属性（逗号分隔的 8 个字段）和 `#text` 文本内容。XML 接口的优点是稳定、弹幕数量固定（B站返回视频的全部弹幕），缺点是弹幕数量可能比 seg.so 接口少（B站可能对 XML 接口做了弹幕筛选或截断）。

**Protobuf 协议（seg.so 新版）**：将视频总时长（duration，秒）除以 360 秒（6 分钟）得到分段数 `segmentCount = min(100, max(1, ceil(duration / 360)))`。上限 100 个分段确保即使面对 10 小时的超长视频也不会发送过量请求。

分段采用串行请求策略（非并行），每段之间强制间隔 2 秒。这是对 B站 API 频率限制机制的适配——实测发现，并行请求 5 个以上分段时，B站服务器会返回 HTTP 429（Too Many Requests）或空响应。串行请求虽然有加载耗时的代价（20 分段 × 2 秒间隔 = 至少 40 秒），但相比并行导致的全量重试或封禁风险，确保了弹幕获取的可靠性。

每个分段的请求和解析结果记录到 `results[]` 数组（按分段索引）。加载完成后统一分析：成功解析且有数据的段直接提取弹幕；失败或返回空数据的段收集到 `retryIndices[]` 列表。等待 5 秒冷却后，对失败段逐一重试（每个失败段之间仍然间隔 2 秒）。重试仍然失败的段被放弃，不影响已成功加载的弹幕数据。

加载进度通过服务端的 `danmuProgressMap`（Map 对象）向客户端实时推送。Map 中存储 `{current, total, phase}` 结构（phase 为 "loading" 或 "retrying"）。客户端通过 `GET /api/danmu/progress?id=xxx` 800ms 间隔轮询，在前端状态栏显示"正在加载弹幕... 15/20"或"正在重试弹幕... 2/5"等文字提示。

#### 4.2.2 腾讯视频弹幕

腾讯视频弹幕采用 30 秒固定长度的分片策略。首先计算总分片数：`totalSegments = Math.ceil(durationMs / 30000)`。以 30 秒为步长（`step = 30000`）遍历分片，构造请求 URL：`https://dm.video.qq.com/barrage/segment/{vid}/t/v1/{start}/{end}`。每片之间设置 10ms 微延迟（`await sleep(10)`），避免请求过快触发腾讯的频率控制。

解析响应中 `barrage_list` 数组的每个条目，提取 `content`（文本）、`time_offset`（毫秒，需除以 1000 转为秒）、`content_style`（JSON 字符串，解析后提取渐变颜色）。颜色处理支持两种格式：`style.gradient_colors[0]`（多级渐变色数组的首色）和 `style.color`（单一颜色值）。颜色值为不含 `#` 的十六进制字符串（如 `ffcc00`），需加 `#` 前缀转换为 CSS 格式。T`content_style` 的 JSON 解析失败时（如字段为空或格式错误），回退为默认白色 `#ffffff`。

单段请求失败时，通过 `try/catch` 捕获异常，使用 `logger.warn()` 记录警告日志，然后继续处理下一个分段。这种"软失败"策略（失败静默跳过）基于一个实际观察：腾讯视频的弹幕分段接口偶尔因服务端负载返回 5xx 错误，但重试通常可以成功。为了不增加加载耗时，由 B站 的 seg.so 重试策略不同，腾讯视频选择了无重试模式。

最终弹幕数组按 `time` 升序排序后返回，确保前端按时间戳顺序渲染。

#### 4.2.3 芒果TV弹幕

芒果TV弹幕接口采用索引递增探测模式，无法预知总段数。接口 URL 为 `https://bullet-ws.hitv.com/bullet/tx/{date}/{time}/{videoId}/{index}.json`，其中 `date` 格式为 `YYYY/MM/DD`，`time` 格式为 `HHMMSS`（6 位时间标识，由用户在前端输入时指定）。

探测流程使用 `while(true)` 无限循环，每次请求一个分段后递增索引 `index++`。终止条件为连续两次检测到 `NoSuchKey` 字符串（`noKeyCount >= 2`），使用连续两次而非一次作为终止条件的目的是容错：芒果TV的 CDN 偶尔会对有效分段返回一次性的 NoSuchKey（可能是缓存未命中），连续两次 NoSuchKey 才确认后续无更多分段。

每个分段的处理逻辑：若响应数据中 `data?.data?.items` 数组存在且非空，解析每条弹幕的 `content` 和 `time`（毫秒转秒），累加到结果数组；若 `NoSuchKey`，`noKeyCount` 递增；若请求异常（网络超时、HTTP 错误等），同样递增 `noKeyCount`（异常也计入 NoSuchKey 计数，因为芒果TV对不存在的分段也返回 NoSuchKey 而非 HTTP 404）。

弹幕缓存文件名使用 `mango_{timeStr}_{videoId}.json`，因为同一个 videoId 在不同时间（不同的 `HHMMSS` 标识）可能有不同的弹幕数据（例如综艺节目的不同期数）。

#### 4.2.4 直播吧弹幕

直播吧的弹幕数据通过分页 API 获取。首先请求计数字段：`https://dan.zhibo8.cc/data/2026/{type}/{matchId}_count.htm?rand={random}` 获取 `{num, per_page}`。`num` 为总弹幕数，`per_page` 为每页数量（通常 5 条）。计算最新页码 `page = ceil(num / per_page) - 1`（0-indexed），然后请求 `https://dan.zhibo8.cc/data/2026/{type}/{matchId}_{page}.htm?rand={random}`。

`rand` 参数（`Math.random()`）用于跳过 CDN 缓存，确保每次请求获取最新数据。

去重机制是本模块的核心设计：服务端完全无状态——不存储任何客户端的 `lastMaxId`。每次请求由客户端传入它记录的 `lastMaxId`，服务端在返回的弹幕数组中筛选 `item.id > lastMaxId` 的条目，同时统计新的 `maxId` 返回给客户端。多客户端独立维护各自的 `lastMaxId`，互不干扰。这种设计使得服务端可以水平扩展（任意实例都能处理请求），无需分布式状态同步。

弹幕按 `room` 字段区分颜色：`room=1`（球迷评论）显示白色（`#ffffff`），`room=2`（彩民下注评论）显示金色（`#ffcc00`），方便用户一眼区分不同群体的评论。

#### 4.2.5 腾讯体育弹幕

腾讯体育是五种弹幕源中实现最复杂的，使用 POST 请求和 cursor 分页机制。

**接口设计**：POST `https://zbpbaccess.video.qq.com/trpc.live_main_logic.live_danmu_read.LiveDanmuRead/PullMsgH5`，请求体为 JSON：

```json
{
  "room_id": 1532819195,
  "cookie": "上次响应的 cursor 值",
  "program_id": "366831"
}
```

固定查询参数 `vappid=34382579&vsecret=e496b057758aeb04b3a2d623c952a1c47e04ffb0a01e19cf`。

**cursor 分页**：与直播吧的页码分页不同，腾讯体育使用服务端驱动的游标（cursor）分页。每次请求携带上一次响应返回的 `cookie` 值（非 HTTP Cookie，而是接口自定义的分页游标字段），服务端根据该游标返回后续的弹幕。首次请求传入空字符串 `""`。

**消息类型过滤**：响应中包含两类消息：`nonconst_speed_msg_info[]`（非固定速度弹幕消息，msg_id=33）和 `const_speed_msg_info[]`（固定速度弹幕/点赞消息，msg_id=48）。系统只提取 `nonconst_speed_msg_info` 中的弹幕，排除点赞消息。每条弹幕的 `danmuku_info` 中包含 `content.content_v2`（文本内容）、`danmaku_control_info.post_time`（发送时间戳）。

**身份认证**：腾讯体育弹幕接口需要有效的腾讯视频登录 Cookie。Cookie 获取方式有两种：一是通过书签脚本（bookmarklet）在腾讯体育页面上执行，从 `document.cookie` 获取 Cookie 字符串并打包为 JSON 粘贴到前端输入框；二是在服务端 `.env` 配置文件中设置 `TXSP_COOKIE` 环境变量，服务端优先使用请求参数中的 cookie，参数为空时回退到环境变量。日志中 cookie 仅记录加密指纹（前 16 字符 + 后 8 字符），完整值仅在 debug 级别可见。

**图片弹幕过滤**：检查每条弹幕的 `res.image_infos` 映射表，若该弹幕的文本内容在此表中存在对应的图片映射，则跳过该弹幕（它是图片弹幕，纯文本渲染无法显示图片内容）。

**自适应轮询间隔**：响应中的 `pull_interval` 字段（毫秒）由服务端动态指定（通常 3000-5000ms），客户端通过 `setTimeout(poll, pullInterval)` 动态调整轮询频率，而非使用固定间隔的 `setInterval`。这种"服务端指导轮询速率"的设计可以响应服务端的流量控制需求。

#### 4.2.6 弹幕源参数获取方案

**B站 BV/EP 号**：通过视频文件名自动识别——正则表达式从文件名中匹配 `BV[0-9A-Za-z]+` 格式的 BV 号或 `ep\d{4,}` 格式的 EP 号。

**腾讯视频 VID**：通过视频文件名自动识别——正则表达式匹配 10-12 位的字母数字组合（排除 BV 开头的字符串）。

**直播吧 matchId**：通过前端输入框手动输入（比赛编号显示在 zhibo8.com 的 URL 中）。

**腾讯体育 room_id/program_id**：通过书签脚本在腾讯体育页面运行后提取。书签脚本（`txsp_bookmarklet.js`）支持三种提取策略：（1）扫描页面全局变量对象中匹配 `/room|program|match|live/i` 的 key；（2）正则扫描所有 `<script>` 标签内容；（3）将 `window.__NUXT__` 对象序列化为 JSON 后进行正则匹配，特别针对 NUXT 闭包参数中相邻字符串数值的模式 `{}, "room_id值", "program_id值"`；（4）若上述策略均未找到，则自动 Hook XHR 请求，拦截 `zbpbaccess` 接口调用并从 POST body 中捕获 room_id 和 program_id。成功提取后自动复制到剪贴板。

### 4.3 Electron 透明悬浮窗实现

#### 4.3.1 窗口创建与管理

窗口创建在 `electron/main.js` 中，核心窗口配置已在前文介绍。对每个显示器独立创建窗口：

```javascript
function createAllWindows() {
  const displays = screen.getAllDisplays();
  for (const display of displays) {
    const win = createOverlayWindow(display);
    overlayWindows.push(win);
  }
}
```

每个窗口的 `bounds`（位置和尺寸）与对应显示器的 `bounds` 完全重合，实现全屏覆盖。

**多显示器热插拔**：通过监听 `screen` 模块的 `display-added` 和 `display-removed` 事件实现。新显示器接入时，自动创建覆盖窗口并加入 `overlayWindows[]` 数组。显示器移除时，通过遍历 `overlayWindows[]` 比较窗口 bounds 与移除显示器的 bounds 来定位对应的覆盖窗口并关闭。这种基于坐标匹配的方法在多个相同分辨率的显示器环境下仍然可靠——因为 Electron 的 `BrowserWindow.getBounds()` 返回的是屏幕绝对坐标，不同显示器的坐标不同。

**可见性心跳**：每 3 秒执行 `win.setAlwaysOnTop(true, 'screen-saver')` + `win.setVisibleOnAllWorkspaces(true, { visibleOnFullScreen: true })`。`'screen-saver'` 级别的置顶高于 macOS 全屏应用和屏幕保护程序，是 Electron 支持的最高置顶级别。

#### 4.3.2 IPC 通信与安全桥接

`electron/preload.js` 使用 `contextBridge.exposeInMainWorld()` 暴露安全 API：

```javascript
contextBridge.exposeInMainWorld('electronAPI', {
  onShortcut: (channel, callback) => {
    const validChannels = [
      'play-pause', 'seek-forward', 'seek-backward',
      'seek-forward-big', 'seek-backward-big',
      'opacity-up', 'opacity-down', 'fontsize-up', 'fontsize-down',
      'toggle-panel', 'load-danmaku'
    ];
    if (validChannels.includes(channel)) {
      ipcRenderer.on(`shortcut:${channel}`, (_event, ...args) => callback(...args));
    }
  },
  setClickThrough: (enabled) => {
    ipcRenderer.send('set-click-through', enabled);
  },
  closeWindow: () => {
    ipcRenderer.send('close-window');
  }
});
```

白名单机制确保前端只能调用预定义的 11 个通道。任何不在白名单中的通道名称（如通过开发者工具注入的恶意通道）的调用都会被静默忽略，防止攻击者通过 XSS 漏洞操纵 Electron 主进程。

#### 4.3.3 点击穿透策略

控制面板的点击穿透控制是悬浮窗用户体验的关键。悬浮窗默认处于穿透模式（`setIgnoreMouseEvents(true, { forward: true })`），用户的所有鼠标操作直接作用于下方播放器。当用户通过 `Cmd+O` 或点击红点唤出控制面板时，调用 `setIgnoreMouseEvents(false)` 临时关闭穿透，面板上的按钮、滑块、输入框可以正常响应鼠标点击和输入。

`{ forward: true }` 参数确保即使在穿透模式下，鼠标事件仍然被转发给操作系统，而不是被 Electron 窗口吞掉。如果没有该参数，底层应用无法收到鼠标事件，用户会感觉"鼠标在那个区域失灵了"。

面板自动隐藏计时器（20 秒无操作）触发后，恢复穿透模式。面板内的任意点击都会重置计时器（通过 `controlPanel.addEventListener('click', resetHideTimer)`），确保用户在操作面板时不会突然被隐藏。

### 4.4 IINA 播放器联动

#### 4.4.1 轮询架构

IINA 联动通过 Python 脚本（`iina-watcher.py`）实现，脚本以独立进程运行（由 `start_overlay.sh` 在后台启动）。轮询间隔为 3 秒。

轮询循环的每个周期执行以下操作：

1. 检查 `/tmp/iina-socket` 是否存在 → 不存在则打印"等待 IINA"并重置状态，跳过本周期
2. 查询 `path` 属性 → 获取当前播放文件路径
3. 查询 `time-pos` 属性 → 获取当前播放秒数
4. 查询 `pause` 属性 → 获取暂停状态
5. 根据状态变化执行相应操作

#### 4.4.2 自动联动矩阵

| IINA 事件 | 检测方法 | 系统响应 |
|-----------|---------|---------|
| 打开新视频 | `path != last_path` | 通知 overlay 切换文件夹、选中视频文件、识别弹幕源、恢复进度、加载弹幕 |
| 播放中 | 每次 `pos != last_pos` | POST /api/progress 保存当前进度 |
| 暂停 | `pause == true && pause != last_paused` | POST /api/iina-state 通知 overlay 暂停弹幕 |
| 恢复播放 | `pause == false && pause != last_paused` | POST /api/iina-state 通知 overlay 恢复弹幕 |
| 快进/快退 | `abs(time - iinaLastTime) > 30 && path == iinaLastPath` | 触发 overlay seek() 跟随跳转 |
| 退出 IINA | Socket 文件消失 | 重置所有状态变量 |

快进检测的 30 秒阈值是一个经验值。常规播放中，两次 3 秒间隔轮询之间的时间差恰好是 3 秒左右（有一定的播放精度误差）。大于 30 秒的跳变几乎可以确定是用户主动的快进或快退操作（如拖动进度条、点击时间线、点击"跳过片头"按钮等）。

#### 4.4.3 文件切换的异步协调

视频文件切换时的自动弹幕源加载是 IINA 联动中最复杂的子流程。切换流程中的核心难点是时间协调：overlay 端加载文件夹列表和视频文件列表是异步的（需要向服务端发送 HTTP 请求），IINA 已经切换到新视频但前端文件列表可能还没就绪。

解决方案是使用 `setInterval` 轮询等待（300ms 间隔，最多 10 次，共 3 秒）。每个间隔检查视频文件列表是否已加载完成（通过查找匹配的视频文件名）。若 10 次轮询后仍未找到（可能用户打开了一个不在配置目录中的视频），则放弃尝试并清除定时器，避免无限等待。

### 4.5 视频流转码

`services/ffmpeg.js` 提供了三种视频流模式的统一入口：

- `streamDirect(videoPath, req, res)`：直接传输，支持 HTTP Range 请求实现随机访问
- `transcodeStream(videoPath, req, res)`：硬件加速实时转码，输出到 HTTP 响应流
- `generateLocalThumb(videoPath, thumbPath)`：本地缩略图生成（在 `/api/thumbnail` 的 fallback 路径中使用，当 B站封面不可用时自动调用）

**直传模式**的关键实现细节是 Range 请求处理。浏览器视频播放器在用户拖动进度条时会发送带有 `Range: bytes=xxx-yyy` 头部的新请求。服务端解析 Range 头部，计算起止字节，使用 `fs.createReadStream(videoPath, { start, end })` 创建区间读取流，设置 206（Partial Content）状态码和 `Content-Range` 响应头，做到"只传输请求的字节范围"。对于不支持 Range 的旧版客户端，回退为全量传输（200 状态码 + `Accept-Ranges: bytes` 头部告知客户端支持范围请求）。

**转码模式**通过 `spawn('ffmpeg', args)` 创建 ffmpeg 子进程，将 ffmpeg 的标准输出（`stdout`）直接管道连接（`pipe`）到 HTTP 响应流（`res`）。`Transfer-Encoding: chunked` 响应头告知客户端响应体是分块传输的（长度未知，持续流式输出）。当客户端断开连接时（关闭页面、停止播放），`req.on('close')` 事件触发，脚本通过 `ffmpeg.kill('SIGKILL')` 强制终止 ffmpeg 进程，避免无用的转码进程继续占用 CPU 和内存资源。

### 4.6 安全与鲁棒性设计

#### 4.6.1 路径穿越防护

所有涉及用户输入的路径参数均经过三层安全校验：

**第一层：`normalizeRelativePath()` 清除 `../` 前缀**。使用正则表达式 `replace(/^(\.\.(\/|\\|$))+/, '')` 移除路径开头的所有"上一级目录"标记。

**第二层：`isPathInside()` 验证路径包含关系**。将候选路径和基准路径都解析为绝对路径（`path.resolve()`），使用 `path.relative(baseDir, candidate)` 计算相对路径。若相对路径以 `..` 开头或为绝对路径，表示候选路径已逃逸出基准目录。

**第三层：`realpathSync()` 解析符号链接**。在目录和文件验证函数（`resolveDirectoryInside`、`resolveFileInside`）中，先验证解析前路径在基准目录内，再通过 `fs.realpathSync()` 获取真实路径（跟随符号链接解析），最后验证真实路径也在基准目录内。这防止了通过符号链接（symlink）绕过路径检查的攻击。

#### 4.6.2 Cookie 安全传输

腾讯体育 Cookie 的传输策略采用"服务端优先"原则：(1) 在 `.env` 中配置 `TXSP_COOKIE` 环境变量（推荐方式，全程 HTTPS 无传输暴露）；(2) 前端通过查询参数传入（仅在未能配置环境变量的情况下使用）。服务端日志中 cookie 以加密指纹形式记录（`前16字符...后8字符`），完整 cookie 仅在 debug 级别日志中可见。

#### 4.6.3 僵尸 Socket 检测

3 秒间隔 × 5 次连续失败 = 15 秒后判定 Socket 为僵尸状态并自动重置。这种设计避免了无限等待，确保 IINA 重启后系统能自动恢复联动。

#### 4.6.4 转码任务中断恢复

服务器重启后，`cache/convert_history.json` 中标记为 `running` 或 `probing` 状态的转换任务自动更新为 `interrupted` 状态。前端查询历史时可以看到任务被中断，决定是否需要重新发起转码。

#### 4.6.5 视频自动清理

系统内置两套自动清理机制。**视频过期清理**：每小时运行一次，读取 `cache/retention_config.json` 中的保留策略（每个文件夹或文件可以配置独立的保留天数），删除超过保留期限的视频文件，同时自动清理已失效的规则条目。**播放记录过期清理**：每小时检查 `playback/` 目录下的播放进度文件，删除超过 `PLAYBACK_MAX_AGE`（环境变量，默认 30 天）未更新的记录，防止播放进度文件无限累积。

### 5.4 系统局限性

1. **B站弹幕加载耗时**：seg.so 串行请求策略导致 2 小时以上的长视频加载时间可能超过 1 分钟。改进方向是引入并发池控制（如 5 并发），在遵守 B站频率限制的同时将加载时间压缩到串行模式的 1/5。

2. **ffmpeg 硬件加速的跨平台限制**：VideoToolbox 是 macOS/iOS 专有框架，Windows 平台需要切换到不同的编码器（Intel QuickSync 或 NVIDIA NVENC），Linux 平台则依赖 VAAPI。当前系统仅在 macOS 上提供了开箱即用的硬件加速支持。

3. **腾讯体育 Cookie 时效性**：Cookie 的有效期通常为几小时到一天，用户需要周期性地通过书签脚本刷新。未来可实现基于 OAuth 2.0 的 token 授权流程，避免手动 Cookie 管理。

4. **弹幕过滤缺失**：当前系统渲染所有获取的弹幕，在弹幕密度极高（数万条/分钟）的视频片段中可能导致视觉拥堵。未来需要实现弹幕密度动态调节和内容去重/过滤功能。

## 第六章 总结与展望

### 6.1 工作总结

本文设计并实现了一套多源弹幕聚合与跨播放器透明悬浮系统，完整涵盖了从弹幕数据获取、数据标准化、前端渲染引擎、桌面窗口封装到播放器联动和跨设备协同的全链路。具体成果包括：

**1. 多源弹幕数据聚合引擎**。设计并实现了五种异构弹幕源的适配器和统一数据模型：Bilibili 双协议（XML + Protobuf，含分段串行加载和失败重试）、腾讯视频分片 API、芒果TV 索引探测、直播吧增量轮询（服务端无状态去重）、腾讯体育 cursor 分页（自适应轮询间隔）。支持 VOD 弹幕缓存、备份和强制刷新。

**2. 播放器无关的弹幕渲染引擎（DanmakuEngine）**。设计并实现了基于轨道调度算法和 requestAnimationFrame 手动动画循环的弹幕渲染引擎。核心优化包括：时间戳驱动的轨道分配算法（O(N) 时间复杂度）、DOM 对象池复用（避免创建/销毁开销）、离屏 Canvas 文本宽度预测量（避免触发 layout）、GPU 加速的 translate3d 变换、每帧发射上限控制。解决了 Electron 透明窗口下 CSS Web Animations 不可靠的工程难题。

**3. Electron 透明悬浮窗系统**。实现了支持多显示器热插拔和 macOS 全屏 Space 兼容的透明悬浮窗系统，包括可见性心跳机制、全局快捷键注册（Cmd+Space/←/→/↑/↓/O/L/Q）、安全 IPC 白名单桥接、点击穿透策略等。

**4. IINA 播放器深度联动**。基于 mpv JSON IPC 协议，使用 Python 轮询脚本实现了播放进度实时同步、暂停/播放自动跟随、快进/快退跳变检测（30 秒阈值）、视频文件切换的自动弹幕源识别与加载。同时设计了僵尸 Socket 检测和恢复机制。

**5. 跨设备协同系统**。通过服务端持久化的 iPad 设置同步（`cache/ipad_settings.json`）和播放进度同步（`playback/` 目录），实现了桌面端与 iPad 移动端的无缝会话漫游。设置变更通过 500ms debounce 自动保存，关键状态在粘贴提取和弹幕加载时立即同步。

**6. 辅助工程功能**。实现了 ffmpeg VideoToolbox 硬件加速实时视频转码、B站番剧 EP 号批量重命名、视频保留策略自动清理、YouTube 视频自动下载（含封面下载和 7 天过期自动删除）等功能，丰富了系统的实用性。

### 6.2 未来展望

**1. 弹幕加载性能优化**。将 B站 seg.so 分段加载从串行改为并发池控制（Channel-based concurrency），在遵守频率限制的前提下最大化并行度。预期将 2 小时视频的弹幕加载耗时从 45 秒压缩到 10 秒以内。

**2. OCR 万能自动同步**。对于不支持 IPC 的播放器（如 PotPlayer、Windows Media Player），通过屏幕截图 + OCR（Tesseract 或 PaddleOCR）识别播放器时间显示区域的数字，实现真正的播放器无关自动同步。这使系统名"外挂"在技术上名副其实——完全不需要播放器配合。

**3. AI 增强弹幕体验**。接入大语言模型（LLM）API，实现三项 AI 能力：(a) AI 实时翻译——对非母语视频的弹幕进行实时翻译；(b) AI 弹幕摘要——对密集弹幕区生成"大家都在讨论什么"的语义摘要；(c) AI 弹幕智能过滤——基于用户偏好（如过滤剧透、情感分类）的动态弹幕筛选。

**4. 社区弹幕共享**。设计弹幕分享协议，允许用户将自己获取的弹幕缓存分享给其他用户，形成去中心化的弹幕分发网络。每个用户的弹幕缓存既服务于本地播放，也可通过 P2P 协议（如 IPFS 或 BitTorrent）共享。

**5. 更多弹幕源和平台支持**。扩展弹幕源至 niconico（日本）、Twitch（全球）、YouTube Live Chat 等国际平台；同时将透明悬浮窗方案推广至 Windows 平台（需解决 WID 透明窗口与 macOS 的差异）和 Linux 平台。
